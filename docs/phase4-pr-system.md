# Phase 4: Pull Request System - Implementation Summary

## Overview

The Pull Request System provides visibility and control over code generated by the builder pipeline. It automatically creates PRs when the builder completes, integrates with the Court review system, and provides a comprehensive dashboard for monitoring and managing code changes.

## Architecture

```
Proposal Approved
      ↓
Builder Pipeline Executes
      ↓
Code Generated & Committed
      ↓
PR Auto-Created (via callback)
      ↓
Court Code Review (future)
      ↓
Maintainer Approval
      ↓
Merge to Main
```

## Components

### 1. Pull Request Model (`internal/pullrequest/`)

**Data Model:**
- **PullRequest**: Core PR structure with metadata, build results, and approval tracking
- **CourtReview**: Court persona feedback on generated code
- **InlineComment**: Line-level code comments (future enhancement)

**Status Flow:**
- `open` → PR is active and awaiting review/approval
- `merged` → PR has been merged to base branch
- `closed` → PR was closed without merging

**Court Review States:**
- `pending` → Waiting for Court review
- `in_progress` → Court is actively reviewing
- `approved` → Court approved the code
- `rejected` → Court rejected the code
- `skipped` → Court review not required (low risk)

### 2. PR Store (`internal/pullrequest/store.go`)

**Filesystem-based Storage:**
- Each PR stored as JSON file in `<data-dir>/pullrequests/`
- Filename: `<pr-id>.json`
- ACID operations with file locking

**Operations:**
- `Create(pr)` - Create new PR
- `Get(id)` - Get PR by ID
- `GetByProposalID(proposalID)` - Find PR for a proposal
- `List(statusFilter)` - List all PRs, optionally filtered by status
- `Update(pr)` - Update PR
- `Approve(prID, approvedBy)` - Mark PR as approved
- `Close(prID)` - Close PR without merging
- `MarkMerged(prID)` - Mark PR as merged
- `AddCourtReview(prID, review)` - Add Court review

### 3. API Handlers

#### Core PR Operations (`cmd/aegisclaw/handlers_pr.go`)

**`pr.list`** - List pull requests
- Optional status filter: `open`, `merged`, `closed`
- Returns summary view for all matching PRs

**`pr.get`** - Get single PR
- Query by `id` or `proposal_id`
- Returns full PR details

**`pr.approve`** - Approve PR for merge
- Requires PR ID and approver name
- Updates approval status and timestamp
- Logs to kernel audit trail

**`pr.close`** - Close PR without merging
- Requires PR ID
- Marks as closed and logs to audit trail

**`pr.merge`** - Merge PR to base branch
- **Validation checks:**
  - PR status must be `open`
  - Build must have passed
  - Analysis must have passed
  - Security gates must have passed
  - Court review must be approved (if required)
  - PR must be approved by maintainer
- Returns detailed error if validation fails
- Logs merge to kernel audit trail

#### Dashboard Handlers (`cmd/aegisclaw/dashboard_pr_handlers.go`)

**`dashboard.pr.list`** - Dashboard-optimized PR list
- Returns formatted data for UI display
- Includes short commit hashes
- Pre-formatted timestamps
- Can merge status computed

**`dashboard.pr.detail`** - Full PR detail view
- All fields from summary
- Complete description
- Court reviews with inline comments
- Approval details

**`dashboard.pr.stats`** - Aggregated statistics
- Total PRs by status
- Court review breakdown
- Ready-to-merge count
- Build/analysis/security pass counts
- Aggregate file changes/additions/deletions

### 4. Auto-Creation from Pipeline (`cmd/aegisclaw/pr_autocreate.go`)

**Pipeline Integration:**
- Builder pipeline calls `onPRCreated` callback after completion
- Callback signature: `func(proposalID, branch, commitHash string, result *PipelineResult)`

**PR Creation Process:**
1. Fetch proposal metadata (title, description, skill)
2. Generate PR title: `[skill-name] Proposal Title`
3. Create description with:
   - Proposal details
   - Build results (files changed, analysis, security)
   - Timeline (start, complete, duration)
4. Set build/analysis/security status from pipeline result
5. Always require Court review for generated code
6. Save to PR store
7. Log creation to kernel audit trail

**Fields Auto-Populated:**
- Title, description (from proposal)
- Branch, commit hash (from pipeline)
- Author: `builder-pipeline`
- Build/analysis/security results
- Files changed count
- Court review required: always `true`
- Court review status: `pending`

### 5. Pipeline Callback Hook (`internal/builder/pipeline.go`)

**Callback Field:**
```go
onPRCreated func(proposalID, branch, commitHash string, result *PipelineResult)
```

**Configuration:**
```go
pipeline.SetPRCreatedCallback(func(proposalID, branch, commitHash string, result *PipelineResult) {
    // Create PR and trigger Court review
})
```

**Invocation:**
- Called at pipeline Step 10 (after successful completion)
- Only called if callback is configured
- Receives full pipeline result for metadata

## Merge Validation

A PR can only be merged if **ALL** of the following are true:

1. ✅ PR status is `open`
2. ✅ Build has passed
3. ✅ Analysis has passed
4. ✅ Security gates have passed
5. ✅ Court review is approved (if required)
6. ✅ PR is approved by a maintainer

If any check fails, the merge is blocked and the API returns an error with specific reasons.

## Audit Trail

All PR operations are logged to the kernel audit trail:

- **`pr.create`** - PR created from pipeline
  - Includes: PR ID, proposal ID, branch, commit, files changed
- **`pr.approve`** - PR approved
  - Includes: PR ID, approver name
- **`pr.close`** - PR closed
  - Includes: PR ID
- **`pr.merge`** - PR merged
  - Includes: PR ID, proposal ID, merger name, branch, commit

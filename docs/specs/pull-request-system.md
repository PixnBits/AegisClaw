# Pull Request System Specification

**Status:** Draft  
**Last Updated:** May 2026
**Related:** `docs/phase4-pr-system.md` (implementation summary), `docs/specs/sdlc-web-portal.md`

## Purpose

The Pull Request System provides visibility, review, and control over code generated by the builder pipeline. It automatically creates PRs upon successful build completion, integrates with the Governance Court for code-level review, and offers a comprehensive interface for monitoring and managing changes.

## Core Principles

- **Transparency**: All generated code changes are visible and reviewable
- **Court Involvement**: Mandatory code review for generated skills (configurable for low-risk)
- **Security First**: Merge validation enforces all security gates before deployment
- **Auditability**: Every PR action is logged to the Merkle audit trail

## Key Features

### 1. Automatic PR Creation
- Triggered automatically after successful builder pipeline completion
- Inherits context from the original proposal
- Pre-populated with build results, security gate status, and SBOM
- Always requires Court review for generated code

### 2. PR Data Model
- **Status**: `open`, `merged`, `closed`
- **Court Review States**: `pending`, `in_progress`, `approved`, `rejected`, `skipped`
- Rich metadata: title, description, branch, commit hash, files changed, build/analysis/security results

### 3. Merge Validation
A PR can only be merged if **ALL** of the following are true:
- PR status is `open`
- Build has passed
- Analysis has passed
- Security gates have passed
- Court review is approved (if required)
- PR is approved by a maintainer

- Court personas review actual generated code (Phase 4 enhancement)

### 5. Dashboard Integration
- PR list and detail views
- Aggregated statistics (by status, Court review, etc.)
- Quick actions (approve, close, merge)

## API Surface

### Core Operations
- `pr.list` (optional status filter)
- `pr.get` (by ID or proposal ID)
- `pr.approve`
- `pr.close`
- `pr.merge`

### Dashboard
- `dashboard.pr.list`
- `dashboard.pr.detail`
- `dashboard.pr.stats`

## Storage
- Filesystem-based JSON store (`<data-dir>/pullrequests/`)
- ACID operations with file locking

## Audit Trail
All operations are logged:
- `pr.create` (from pipeline)
- `pr.approve`
- `pr.close`
- `pr.merge`

## Future Enhancements
- Inline code comments with resolution
- Threaded discussions on PRs
- Real-time build status updates (SSE)
- Deployment approval workflow tied to PR merge

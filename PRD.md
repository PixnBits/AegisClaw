### 1. Overview & Mission

SeedClaw is a **minimal, local-first, self-bootstrapping, zero-trust AI agent platform**.  
After one `go build` and `./seedclaw --start`, the user has a working paranoid agent swarm. The Trusted Computing Base (TCB) is deliberately tiny: **only `seedclaw.go` + the four committed core skills**. Everything else is generated by the `coder` skill, compiled and tested inside a fresh ephemeral sandbox, then registered with **explicit least-privilege mounts only**.

**Tagline:** “Bootstrap your own paranoid agent swarm from markdown prompts only. No cloud. No binaries from strangers. Audit everything.”

All communication flows exclusively through the `message-hub`. No skill ever touches the host filesystem or another skill’s data.

---

### 2. Core Principles (MUST – non-negotiable)

1. **Minimal committed core, zero-code for everything else**  
2. **Sandbox-first, not sandbox-later** – every skill runs in its own Docker container with the isolation profile defined in ARCHITECTURE.md § Sandbox & Isolation.  
3. **Reliable self-bootstrapping loop** – seedclaw is the only binary that runs on metal and is the sole editor of `compose.yaml`.  
4. **Explicit Host-Skill Boundary** – **never** mount the entire `./shared/` directory. Each skill receives **only** the purpose-named subdirectories it explicitly declares in its registration metadata (ro/rw flags enforced by seedclaw).  
5. **Single communication gateway** – `message-hub` is the only component allowed to talk to the host control socket. All skill↔skill and skill↔host traffic is JSON-routed through it.  
6. **Immutable audit trail** – every significant action performed by seedclaw is appended to `shared/audit/` with cryptographic hash chaining (future) and human-readable JSON lines.

### 3. Functional Requirements

#### 3.1 Seed Binary (`src/seedclaw.go`) – MUST
- Single static Go binary (no external runtime except Docker).
- On `--start`:
  - Verify Docker daemon (rootless preferred).
  - Create `./shared/` tree with correct ownership/permissions.
  - Create control socket proxy (loopback TCP or protected Unix socket – **never** exposed directly to generic skills).
  - Start the four core services from `compose.yaml` with **selective mounts only**.
- Maintain persistent skill registry (`~/.seedclaw/registry.json` – versioned, signed on write).
- Accept user input (stdin/stdout loop minimum; optional WebSocket or Telegram via env var).
- Route **every** request through message-hub.
- Skill lifecycle (MUST implement exactly):
  1. Receive “generate_skill” request.
  2. Forward to `coder` skill via message-hub.
  3. Receive proposed skill bundle (Go code + Dockerfile + **registration metadata**).
  4. **Sandbox compile & test** in temporary container (`--network=none`, 30 s timeout, cgroup 512 MiB).
  5. If `go vet` + basic static analysis + smoke test pass → append service block to `compose.yaml` with **only** the declared mounts.
  6. `docker compose up -d --no-recreate`.
  7. Register skill and append immutable audit entry.
- All Docker operations performed **only** by seedclaw (never by any skill).

#### 3.2 Message-Hub (core skill)
- Sole IPC router.
- Receives only the control socket mount (`/run/sockets/control/seedclaw.sock`).
- Enforces structured JSON messages with `from`/`to` fields (schema validated).
- Logs every routed message to central audit trail.

#### 3.3 Core Bootstrap Skills
- Must match their respective `SKILL.md` contracts (llm-caller, ollama, coder).
- All use the exact same Docker isolation profile.
- Coder skill **must** output registration metadata including:
  ```json
  {
    "required_mounts": [
      {"path": "sources", "mode": "ro"},
      {"path": "builds", "mode": "rw"}
    ],
    "resource_profile": {"memory_mb": 512, "timeout_sec": 30},
    "capabilities": ["code_generation"],
    "network_needed": false
  }
  ```

#### 3.4 Generated Skill Lifecycle
Every generated skill **MUST** declare its exact mount requirements. Seedclaw **MUST** enforce them. No defaults, no blanket mounts.

### 4. Security & Isolation Requirements (MANIACAL FOCUS)

- **Default container profile** (hard-coded in seedclaw and enforced on every `docker run`):
  - Ephemeral (`--rm`).
  - `--read-only` + tmpfs `/tmp`.
  - `--network=none` (explicit opt-in only).
  - `--cap-drop=ALL --security-opt no-new-privileges`.
  - Strict seccomp profile (default Docker + extra syscall blacklist).
  - cgroup2 limits (CPU burst 1.0, memory 512 MiB).
  - 30-second hard timeout (kill + audit).
- **Mount strategy** – purpose-driven only. Seedclaw parses metadata and writes **exactly** those volume lines into `compose.yaml`. Any attempt to request `/` or host paths is rejected at generation time.
- **Control channel** – seedclaw runs a minimal IPC proxy bound to loopback. Message-hub connects over Docker network (mTLS or short-lived token in future).
- **Code vetting pipeline** (mandatory before registration):
  - `go vet`, `go test -c` (compile-only), staticcheck (if available in sandbox).
  - Prompt injection guard: reject any code containing `unsafe`, `syscall`, `os/exec` unless explicitly whitelisted in metadata.
- **Threat model** – see ARCHITECTURE.md § Threat Model. PRD adds: prompt injection → sandbox compile rejection; container escape → seccomp + no caps; rogue skill seeing other data → prevented by selective mounts; compose.yaml tampering → only seedclaw edits it and creates backups with SHA-256.

**Auditability invariant:** Every action that touches the host (compose edit, registry write, mount addition) is performed **exclusively** by the seedclaw binary and logged immutably before the action completes.

---

### 5. Auditing & Observability (EASY AUDITING)

- Append-only JSON lines in `shared/audit/seedclaw.log`:
  ```json
  {"ts":"2026-03-11T12:53:00Z","actor":"seedclaw","action":"register_skill","skill":"git-tool","mounts":["sources:ro"],"hash":"sha256:...","status":"success"}
  ```
- Registry JSON is versioned and includes previous hash.
- `compose.yaml` backups on every edit (`compose.yaml.bak.<unix_ts>`).
- All logs centralized; skills never write outside their declared outputs/ mount.

---

### 6. Non-Functional Requirements

- **Performance**: < 2 s latency for simple message routing; skill startup < 5 s.
- **Reliability**: Docker operations retried 3× with exponential backoff; ephemeral containers guarantee clean state.
- **Maintainability**: All skills self-document via `SKILL.md` + registration metadata. Seedclaw code must remain < 10 KB and fully auditable.
- **Portability**: Linux/macOS/Windows (Docker Desktop); amd64/arm64.
- **Resource usage**: Seedclaw binary < 20 MB RAM when idle.

---

### 7. MVP Scope & Success Criteria

**First-run success (must work after `go build`):**
1. `./seedclaw --start` → four core containers running.
2. Paste bootstrap-prompt.md → coder skill generates first new skill.
3. New skill registers, starts, and appears in `docker compose ps`.
4. Audit log contains every step with hashes.

**Security smoke tests** (must pass):
- Attempt to request blanket mount → rejected.
- Skill tries network without declaration → fails.
- All control traffic visible only through message-hub.

---

### 8. Non-Goals (MVP)

- Multi-user / auth / RBAC.
- Persistent state beyond registry + audit.
- GUI / web dashboard.
- Cloud deployment.
- Automatic skill revocation (defer to generated capability).

---

### 9. Minimal Dependencies & Tech Stack

- Go 1.23+ stdlib only where possible.
- `github.com/docker/docker/client` (pinned version for auditability).
- `github.com/ollama/ollama` Go client (inside llm-caller skill).
- Telegram-bot-api or gorilla/websocket only if env vars present.
- No other external binaries or pip/npm.
